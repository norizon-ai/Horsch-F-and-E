  - Convert the notebook to a Python script and run the resulting script:
    ```bash
    jupyter nbconvert --to script /path/to/notebook.ipynb
    python /path/to/notebook.py
    ```
- To run a Jupyter Notebook interactively, use the JupyterHub service provided by the HPC center.
- Consult with colleagues or advisors to optimize memory usage for the job.

- The user was advised to read the output file generated by the job to diagnose errors.

 # HPC Support Ticket: Error in HPC Server Connection

## Subject
Error in HPC server connection while loading LLM model from Hugging Face.

## User Issue
The user is facing a connection timeout error while trying to load a language model from Hugging Face. The code works fine on Kaggle but has been failing on the HPC server for the past two days.

## Error Message
```
Downloading shards:   0%|          | 0/4 [00:00<?, ?it/s]
Downloading shards:   0%|          | 0/4 [00:40<?, ?it/s]
Traceback (most recent call last):
```

## Root Cause
The root cause of the problem is a lack of internet connection without proxies on the HPC server.

## Solution

3. **Alternative Tools**: Consider using alternative tools like LitGPT or Axolotl, which may have better support for the HPC environment.
4. **Resource Allocation**: For testing purposes, use more available resources like A40s instead of A100 80 GB cards.

## Additional Recommendations
- **Model Support**: Ensure that the model being used supports the available resources on the HPC server.
- **Environment Setup**: Verify that the Python environment is correctly set up with all necessary dependencies and configurations.

## Conclusion
The issue is related to network connectivity and can be resolved by configuring the proxy settings correctly. Additionally, using workspaces and alternative tools can help in managing resources more efficiently.

 # Webserver auf GPU Node

## Keywords
- Webserver
- GPU Node
- SSH-Tunnel
- Private IPs
- FAU-Netze
- Proxy
- Sicherheitsrisiko
- Nutzungsbedingungen
- Reverse-Proxy
- Compute-Node

## Problem
Der Benutzer möchte einen Webserver (z.B. für eine Gradio UI) auf einem GPU Node laufen lassen, war aber unsicher, ob dies möglich ist, da er annahm, dass eine Internetverbindung von der GPU Node aus prinzipiell nicht möglich ist.

## Ursache
Die Compute-Nodes haben private IPs, die nur innerhalb der FAU-Netze erreichbar sind und nicht geNATet werden. Sie können nur via Proxy ins Internet und sind selbst aus dem Internet nicht erreichbar.

## Lösung
1. **SSH-Tunnel**: Um einen auf einem Compute-Node laufenden Webservice zu erreichen, muss ein SSH-Tunnel via `csnhr.nhr.fau.de` zum spezifischen Port auf den aktuellen Compute-Node aufgebaut werden.
   ```bash
   ssh -J b116ba19@csnhr.nhr.fau.de -L 8888:127.0.0.1:8888 b116ba19@a0601
   ```
   Hierbei wird angenommen, dass der Webservice lokal auf dem Knoten `a0601` und Port `8888` läuft und lokal via `http(s)://localhost:8888` zugreifbar ist.

2. **Sicherheitsrichtlinien**: Dinge wie "ngrok", "Pagekite", etc., die einen "unkontrollierten" Reverse-Proxy ins Internet aufbauen, sind ein klarer Verstoß gegen die Nutzungsbedingungen der FAU und stellen ein Sicherheitsrisiko dar.

## Dokumentation
Es gibt keine spezifische Dokumentation für das Starten eines Webservers auf einem GPU Node, aber die oben beschriebene Methode mit dem SSH-Tunnel kann verwendet werden, um den Webservice zu erreichen.

## Zusammenfassung
Das Starten eines Webservers auf einem GPU Node ist möglich, aber die Erreichbarkeit ist aufgrund der privaten IPs und der Proxy-Beschränkungen eingeschränkt. Ein SSH-Tunnel kann verwendet werden, um den Webservice zu erreichen. Es ist wichtig, die Sicherheitsrichtlinien und Nutzungsbedingungen der FAU zu beachten.
