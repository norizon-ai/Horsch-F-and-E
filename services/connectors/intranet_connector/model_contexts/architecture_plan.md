# Architectural Plan: Crawler-to-Database Pipeline

## 1. High-Level Goal

To design a robust, scalable, and decoupled pipeline that takes the content generated by the `IntranetCrawler`, processes it, and stores it in the appropriate databases (PostgreSQL for metadata and a vector database for searchable content).

---

## 2. Core Components

The system will be composed of four main components:

1.  **The `IntranetCrawler` (Producer):** Its sole responsibility is to crawl pages and produce `RawArticle` objects (using structured Pydantic models). Instead of being processed in the main script, this data will be sent to a message queue.
2.  **A Message Queue (e.g., RabbitMQ):** This acts as a durable, intermediate buffer. It decouples the crawling process from the data processing and storage, making the system more resilient and scalable. If the database is slow or temporarily down, the crawler can continue its work without losing data.
3.  **An Ingestion Worker (Consumer):** A separate, long-running process that listens for messages from the queue. Its job is to:
    *   Pull `RawArticle` objects from the queue.
    *   Generate vector embeddings for the content.
    *   Save the data to the databases.
4.  **The Database (Storage):**
    *   **PostgreSQL with pgvector:** Instead of two separate databases, we will use a single PostgreSQL instance with the `pgvector` extension. This simplifies the architecture, reduces operational overhead, and allows for powerful queries that combine structured metadata and semantic vector search in a single transaction.

---

## 3. Detailed Data Flow

Here is the step-by-step journey of a single crawled page:

1.  **Crawling:** The `IntranetCrawler` runs with advanced content processing, but instead of `yield`ing `RawArticle` objects to the `main` function, its `crawl` method will now publish each `RawArticle` object (with structured Pydantic validation) as a message to a specific queue in RabbitMQ (e.g., a queue named `crawled_content`).

2.  **Queuing:** The `RawArticle` message sits in the `crawled_content` queue, waiting to be processed. This allows the crawler to work at its own pace, and we can even run multiple crawlers in the future, all feeding into the same queue.

3.  **Consumption:** The `Ingestion Worker` pulls a message from the queue. This is a blocking operation, so the worker waits until a message is available.

4.  **Processing (Embedding Generation):**
    *   The worker takes the `content` (the cleaned markdown) from the `RawArticle` object.
    *   It uses a sentence-transformer model (e.g., from the `sentence-transformers` library) to convert this text into a numerical vector (an embedding). This embedding represents the semantic meaning of the content.

5.  **Storage:**
    *   The worker connects to the PostgreSQL database. In a single atomic transaction, it will:
        *   Insert a new record into the `knowledge_articles` table with the structured information (`source_uri`, `title`, `metadata`, `permissions`).
        *   Store the original `content` text and its newly generated `embedding` vector in the same table.

6.  **Acknowledgement:** Once the data is successfully saved in the database, the worker sends an "acknowledgement" (ack) back to RabbitMQ. This tells the queue that the message has been processed successfully and can be safely deleted.

7.  **Error Handling:** If any step in the processing or storage fails, the worker will *not* send an acknowledgement. RabbitMQ will then automatically requeue the message to be tried again later, preventing data loss.

---

## 4. Proposed Database Schema

*   **PostgreSQL Table: `knowledge_articles`**
    *   `id`: `SERIAL PRIMARY KEY`
    *   `source_uri`: `TEXT UNIQUE NOT NULL`
    *   `source_type`: `TEXT NOT NULL` (e.g., "INTRANET")
    *   `title`: `TEXT`
    *   `content`: `TEXT` (to store the cleaned markdown)
    *   `embedding`: `vector(384)` (assuming `all-MiniLM-L6-v2` model; dimensions must match the model)
    *   `metadata`: `JSONB` (to store status codes, redirects, etc.)
    *   `permissions`: `TEXT[]` (an array of strings)
    *   `created_at`: `TIMESTAMP WITH TIME ZONE DEFAULT NOW()`
    *   `updated_at`: `TIMESTAMP WITH TIME ZONE DEFAULT NOW()`

---

## 5. Key Technologies to Use

*   **Message Queue:** **RabbitMQ** with the `aio-pika` Python library for asynchronous communication.
*   **Embedding Generation:** The `sentence-transformers` library.
*   **Database Interaction:** Elasticsearch database for content and postgresql for user data.

## 6. Current Crawler Implementation

The `IntranetCrawler` has been enhanced with the following capabilities:

### 6.1. Data Models
*   **`RawArticle`**: Structured Pydantic model containing:
    *   `source_document_id`: Unique identifier (typically the URL)
    *   `content`: Clean markdown content extracted from the page
    *   `source`: `RawArticleSource` with URI, module name, and retrieval timestamp
    *   `author`: Optional `RawArticleAuthor` with name
    *   `tags`: List of extracted tags/keywords
    *   `permissions`: List of permission strings
    *   `metadata`: Dictionary with additional metadata (title, status codes, etc.)

### 6.2. Content Processing
*   **Markdown Generation**: Uses `DefaultMarkdownGenerator` from crawl4ai
*   **Content Filtering**: Excludes unwanted HTML elements (forms, headers, footers, navigation)
*   **Link Filtering**: Excludes external and social media links
*   **Deep Crawling**: Configurable BFS/DFS/Best-First strategies with domain filtering

### 6.3. Configuration
The crawler is configured via `CrawlerSettings` in `config.py`:
*   `STRATEGY`: Crawling strategy ("bfs", "dfs", "best_first")
*   `MAX_DEPTH`: Maximum crawl depth
*   `MAX_PAGES`: Maximum pages to crawl
*   `EXCLUDED_TAGS`: HTML tags to exclude from content
*   `EXCLUDE_EXTERNAL_LINKS`: Filter external links
*   `EXCLUDE_SOCIAL_MEDIA_LINKS`: Filter social media links
*   `USE_PLAYWRIGHT`: Enable browser-based crawling
*   `AUTH_TOKEN`: Optional authentication token
